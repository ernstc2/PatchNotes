   - Run `npx @better-auth/cli generate --config src/lib/auth.ts` (or the appropriate CLI flag)
   - The CLI will output a schema file — save it as `src/lib/db/schema/auth-schema.ts`

   If db:migrate fails due to missing DATABASE_URL in local env, use `npx drizzle-kit push` instead (it reads from .env.local).

8. Verify with `npm run build` — TypeScript compilation must pass.

1. Create/finalize `src/lib/auth.ts`:
   ```typescript
   import { betterAuth } from "better-auth";
   import { drizzleAdapter } from "better-auth/adapters/drizzle";
   import { db } from "@/lib/db";
   import * as schema from "@/lib/db/schema";

   export const auth = betterAuth({
     database: drizzleAdapter(db, {
       provider: "pg",
       schema: { ...schema },
     }),
     emailAndPassword: {
       enabled: true,
     },
   });
   ```
   Note: Pass the full flattened schema object so better-auth can find user, session, account, verification tables.

2. Create `src/lib/auth-client.ts`:
   ```typescript
   import { createAuthClient } from "better-auth/react";
   export const authClient = createAuthClient({
     baseURL: process.env.NEXT_PUBLIC_APP_URL,
   });
   ```
   Ensure `NEXT_PUBLIC_APP_URL` is documented as needed env var (e.g., `http://localhost:3000` for dev).

3. Create `src/app/api/auth/[...all]/route.ts`:
   ```typescript
   import { auth } from "@/lib/auth";
   import { toNextJsHandler } from "better-auth/next-js";
   export const { GET, POST } = toNextJsHandler(auth);
   ```

4. Create `proxy.ts` at project root:
   - Export `async function proxy(request: NextRequest)` that checks session via `auth.api.getSession({ headers: await headers() })`
   - If no session, redirect to `/sign-in`
   - Otherwise `NextResponse.next()`
   - Export `config` with matcher: `["/profile/:path*", "/onboarding/:path*"]`

5. Create `src/features/auth/queries.ts` with server actions and queries:
   - `"use server"` directive at top
   - `toggleBookmark(policyItemId: string)` — checks session, then SELECT existing -> if exists DELETE, else INSERT with onConflictDoNothing
   - `getBookmarkedItems(userId: string)` — innerJoin bookmarks with policyItems, ordered by bookmark createdAt desc
   - `getBookmarkedIds(userId: string)` — returns array of policyItemId strings for the user (lightweight query for feed page bookmark state)
   - `isBookmarked(userId: string, policyItemId: string)` — returns boolean for a single item
   - `saveTopics(topics: string[])` — checks session, DELETE all existing user_topics for user, INSERT new rows, then calls `redirect("/")`. Used by onboarding flow (redirects to homepage).
   - `updateTopics(topics: string[])` — same DB logic as saveTopics (delete all + insert new), but calls `revalidatePath("/profile")` instead of redirect. Used by profile page watchlist management (keeps user on /profile).
   - `getUserTopics(userId: string)` — returns array of topic strings for user
   - All mutations verify session via `auth.api.getSession({ headers: await headers() })` and throw if not authenticated

6. After implementing queries.ts, promote the Wave 0 test stubs in `src/features/auth/queries.test.ts` to real tests:
   - Mock `auth.api.getSession` to return a fake session with a test userId
   - Mock `db` (or the drizzle query builder) to verify correct SQL operations
   - Convert each `test.todo()` to a real `test()` that exercises the function and asserts behavior
   - Run `npm test -- src/features/auth/queries.test.ts` to confirm all 6 tests pass

7. Verify: `npm run build` must pass. The auth API route should be reachable (though full testing requires sign-up UI in Plan 02).
